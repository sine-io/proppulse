# Dashboard 单滚动条与品牌化滚动条样式设计

日期：2026-04-05

## 1. 目标

修正当前 Dashboard 页面右侧出现双滚动条的问题，并把保留下来的唯一滚动条做成与页面整体一致的 SaaS 风格样式。

本次改动目标分成两部分：

- 页面只保留一个有效纵向滚动容器
- 该滚动条视觉上与当前 `slate / emerald` 品牌风格统一

## 2. 当前问题

当前页面在视觉上会出现两层纵向滚动：

- 页面外层仍可滚动
- 主内容区 `main` 也可滚动

结果是用户在页面最右侧会看到多余的滚动条，破坏整体观感，也让交互焦点不明确。

## 3. 设计结论

### 3.1 滚动容器职责

保留 `main` 作为唯一纵向滚动容器。

具体职责固定为：

- 根容器：固定到视口高度，并禁止页面外层滚动
- `body`：不承担滚动职责
- `#root`：只负责占满高度
- `main`：承担页面正文滚动

这意味着：

- 页面最右侧只出现一条滚动条
- Sidebar 与 TopHeader 始终固定
- 正文区域独立滚动，符合 Dashboard 使用预期

### 3.2 滚动条视觉方向

采用“品牌化但不过度显眼”的 SaaS 风格滚动条。

推荐样式：

- 宽度：细窄
- 轨道：深 `slate`
- 滑块：中深 `slate` 基底
- hover：轻微提升到带 `emerald` 倾向的亮度
- 圆角：完整圆角

目标效果不是把滚动条做成强视觉组件，而是让它看起来像页面设计系统的一部分。

## 4. 推荐方案

采用“仅给主内容滚动容器加品牌化滚动条样式”的方案。

具体做法：

1. 页面布局层面禁止根页面滚动
2. 保持 `main` 的 `overflow-y-auto`
3. 在样式层只为主内容滚动容器定义滚动条样式
4. 同时覆盖：
   - WebKit 滚动条伪元素
   - Firefox `scrollbar-color`

原因：

- 影响面最小
- 不会误改 Sidebar、表格内部、浏览器默认控件等其他潜在滚动区域
- 更容易验证“只有一个主滚动条”

## 5. 非目标

本次不做：

- 新增复杂滚动动画
- 自定义滚动条拖拽逻辑
- 改动页面布局结构
- 改动数据或组件拆分
- 改动移动端滚动交互

## 6. 实现边界

预计涉及：

- `site/src/App.tsx`
  - 确保根容器与内容容器的高度和 overflow 职责清晰
- `site/src/styles.css`
  - 新增主内容滚动条样式
- `tests/site/app.test.tsx`
  - 增加最小结构断言，锁定“页面外层不滚、主内容区滚动”

## 7. 样式规范

### 7.1 主内容滚动容器选择

滚动条样式只应作用于主内容滚动容器。

不要写成全局 `body::-webkit-scrollbar` 样式，因为这会：

- 把样式错误扩散到不该滚动的区域
- 难以保证只有一个滚动条

### 7.2 配色

推荐配色策略：

- 轨道：`slate-900` / `slate-800` 之间
- 滑块默认：`slate-600`
- 滑块 hover：向 `emerald-400` 或 `emerald-500` 靠近，但保持克制

### 7.3 尺寸

- 滚动条宽度：约 10px 到 12px
- 内边距感通过透明边框或 background-clip 营造
- 拖拽块圆角保持大圆角

## 8. 验证方式

实现完成后至少验证：

```bash
npm run test -- tests/site/app.test.tsx
npm run typecheck
npm run build:site
```

验证标准：

- 单元测试能证明根容器禁滚动、`main` 为滚动容器
- 页面构建通过
- 页面最右侧只保留一个纵向滚动条
- 滚动条视觉风格与当前 Dashboard 一致

## 9. 最终结果

本次改动完成后，Dashboard 页面应满足：

- 右侧不再出现双滚动条
- 只保留主内容区滚动
- 滚动条与整体 `slate / emerald` 品牌视觉一致
- 不引入额外布局回归
