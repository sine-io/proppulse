# Sidebar 功能栏补全设计

日期：2026-04-05

## 1. 目标

补全 Dashboard 左侧功能栏里仍然偏占位的两个入口：

- 房源全库
- 系统设置

让 5 个侧边栏菜单都对应清晰、可见、语义一致的内容区，而不是只停留在锚点跳转或占位文案。

## 2. 当前现状

当前左侧导航共有 5 项：

- 首页
- 重点关注小区
- 房源全库
- 降价雷达
- 系统设置

其中：

- 首页：语义完整，对应页面整体概览
- 重点关注小区：已经有独立专区
- 降价雷达：已经有图表区
- 房源全库：当前只是落到“底部区”，语义不够准确
- 系统设置：当前只是左下角“今日策略”卡片，不是设置区

因此问题不是点击失效，而是“菜单标签”和“内容区语义”不一致。

## 3. 设计结论

采用“单页锚点导航 + 独立专区补全”方案，不引入路由系统。

### 3.1 导航结构

保留当前单页布局和 hash 导航方式。

左侧菜单对应关系调整为：

- 首页 → `#overview`
- 重点关注小区 → `#focus-communities`
- 房源全库 → `#inventory`
- 降价雷达 → `#price-radar`
- 系统设置 → `#settings`

其中：

- `#inventory` 不再只是“底部区”的技术锚点，而是一个真实的“房源全库”专区
- `#settings` 不再复用 Sidebar 内卡片，而是在主内容区提供真实设置卡片区

### 3.2 房源全库专区

新增“房源全库”专区，放在“重点关注小区”之后、“降价雷达”之前不合适，因为全库更像运营/盘点内容；推荐放在“重点关注小区”之后、“底部双栏”之前，形成：

1. 概览
2. 图表
3. 重点关注小区
4. 房源全库
5. 系统设置

该区块展示“当前监控范围内所有小区的摘要总览”，而不是单条房源明细页。

每张卡片展示：

- 小区名称
- 行政区
- 监控状态（正常监控 / 待复核）
- 主监控户型
- 最新挂牌中位价
- 最新挂牌套数
- 最新结论
- 数据来源标签

这样“房源全库”至少能承担“监控对象总览”的角色，和“重点关注小区”形成区别：

- 重点关注小区：强调人工优先关注和摘要判断
- 房源全库：强调全部监控对象的一览盘点

### 3.3 系统设置专区

新增“系统设置”专区，放在页面最下方，作为运营说明和系统状态区。

该区块展示只读型设置/说明卡片，不引入真实表单提交。

包含 3 组信息：

1. **数据刷新**
   - 最近更新时间
   - run artifact 读取策略
   - 报表 / public data 的生成说明

2. **数据来源**
   - 当前支持的数据源类别
   - active / pending_verification 的口径说明
   - 缺失数据为何显示“暂无”

3. **验证与维护**
   - 构建命令
   - E2E 预览命令
   - 当前页面导航与空态策略说明

这个区块的目标不是做后台配置，而是让“系统设置”菜单有真实落点，并对当前站点行为给出可读的系统说明。

## 4. 方案对比

### 方案 A：继续沿用现有锚点，只改文案

优点：

- 改动最小

缺点：

- 用户点击“房源全库”“系统设置”后仍然觉得内容名不副实
- 没有真正补齐功能结构

### 方案 B：新增单页专区（推荐）

优点：

- 保持当前架构不变
- 侧边栏 5 项都能对应明确内容
- 改动集中在首页组装和展示组件，风险低

缺点：

- 需要新增两个展示组件和对应测试

### 方案 C：直接做多页面/路由

优点：

- 长期扩展空间更大

缺点：

- 超出当前单页 Dashboard 范围
- 对现有实现来说改动过重

## 6. 数据策略

### 6.1 房源全库

基于现有真实数据构建，不新增新接口。

来源：

- `communities`
- `primarySegmentsByCommunityId`
- `latestReport`
- `communitySeries`

展示规则：

- 最新挂牌中位价：优先 `latestReport`，回退 `communitySeries`
- 挂牌套数：优先 `latestReport`，回退 `communitySeries`
- 最新结论：优先 `latestReport`，缺失则显示“待观察”
- 数据来源：直接使用社区配置里的 `sourceProvider`

### 6.2 系统设置

来源仍为首页已有 view-model 输入：

- `lastUpdatedLabel`
- `runArtifacts.length`
- `communities.length`
- 各社区状态与 source provider

系统设置区不修改数据，只解释系统状态。

## 7. 测试要求

至少覆盖：

- 点击“房源全库”后看到对应标题和内容
- 点击“系统设置”后看到对应标题和说明卡片
- view-model 能构建房源全库摘要和系统设置摘要
- 页面无数据时仍保持可读空态

## 8. 非目标

本次不做：

- 真正的房源搜索页
- 房源列表筛选器
- 可编辑的系统设置表单
- 路由改造
- 后端接口或数据库
